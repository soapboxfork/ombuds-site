Experiment with Ombuds

For an end user who just wants to download something, we have one piece of software for you to test out. If you run a Unix OS you can (theoretically) download and build a desktop client from Github. If you are an android developer and want a first look at what we have then sign up for the alpha release here.

This is what Ombuds looks like running on a desktop.

There are three ways to interact with Ombuds, each method has a different trust model.

Desktop

Trust Model: Trust our software and your OS.

The desktop client is really a Bitcoin full node, and thus receives every Bitcoin transaction and every Bitcoin block. The client parses all this information, extracts Ombuds bulletins, and indexes them locally. Every Ombuds statement ever published in the block chain can be read using the desktop client. This provably complete record comes at the cost of disk space, processing power, and bandwidth.

The desktop client has a Bitcoin wallet and an Ombuds messenger. The client verifies inclusion of Ombuds bulletins in the block chain.

Mobile

Trust Model: Trust your Bitcoin peers and your phone.

The mobile client includes an SPV client with bloom filters, thus receives only relevant Bitcoin transactions and every Bitcoin block header. Trusting your Bitcoin peers to honor bloom filters reduces total bandwidth usage and recording only block headers reduces total disk space. Unlike the desktop client, the mobile client does not have an index of all Ombuds statements.

This means that a user's peers could deny relevant information to the device and even prevent it from publishing if it had full control over the devices ability to connect to "honest" peers. These compromises are necessary for mobile devices that cannot run the full node.

The mobile client has a full featured Bitcoin wallet and an Ombuds messenger. The mobile client confirms block chain inclusion of Ombuds bulletins, but again it is reliant on peers honoring bloom filters. In practice however, peers do honor bloom filters. Here are some nice pictures:

Web Relay

Trust Model: Trust the operator of a website and HTTPS.

Someone running an Ombuds Desktop Client possesses an index of all Ombuds statements. This index can be formatted and delivered as a web page, viewable to anyone with a browser and an uncensored Internet connection.

However, the person viewing Ombuds bulletins must completely trust the web relay to not censor content. This option provides ultimate convenience: zero disk space, near zero processing power, and near zero bandwidth to see Ombuds bulletins. But a provably complete record of bulletins is in the hands of the website operator.
