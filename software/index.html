Experiment with Ombuds

For an end user like yourself we have two peices of software for you to test out. If you have an android phone then sign up for the alpha release here. If you run a Unix OS you can (theoretically) download and build a desktop client from Github.

A one sentence description:

Ombuds creates and retrieves public statements that are stored in the Bitcoin block chain.

A three sentence description:

Bitcoin provides a resilient peer-to-peer network and a transparent, verifiable ledger. The Ombuds Toolkit defines a message protocol (known as Bulletins) and provides mobile, desktop, and web clients. Together, Ombuds provides a familiar microblogging interface where Bitcoin concepts have been abstracted, yet capitalize on the advantageous properties of Bitcoin.

A four paragraph description:

Ombuds enables any holder of Bitcoin to publish immutable statements within Bitcoin's block chain. These statements, which we call bulletins, are encoded within Bitcoin transactions which can be sent from a smartphone or a computer. These statements can be viewed from any computer that can connect to the Bitcoin network.

If a Bitcoin user can get a transaction into the block chain it is effectively permanent. Ombuds is based off of this principle. Any software that understands Ombuds' messaging format and has a copy of the block chain can then decode these bulletins and display them for the public to view.

To censor a user's bulletins, an attacker must either block access to Bitcoin completely or generate blocks faster than the entire honest network. Since most small nations do not have the hashing infrastructure to mount a 51% attack against the Bitcoin network, Ombudsâ€™s method for storing bulletins is robust.

In essence, we have re-purposed Bitcoin as an immutable public record of free speech and we have built the tools that let anyone publish, view, and interact with these records. This system does not depend on the honesty or availability of any single website, service or hard drive. Instead, Ombuds uses the global distribution of the block chain to ensure the availability of the public record.

Three components comprise the Ombuds system, each with a different trust model.

Desktop

Model: trust yourself.

The desktop client includes a Bitcoin full node, thus receives every Bitcoin transaction and every Bitcoin block. The client parses all this information, extracts Ombuds bulletins, and indexes them locally. Every Ombuds statement ever published in the block chain can be read using the desktop client. This provable transparency comes at the cost of disk space, processing power, and bandwidth, but whose price is minimal for modern machines.

The desktop client has a Bitcoin wallet and an Ombuds message formatter. These are presented to the user in a familiar "create public statement" interface, akin to publishing a tweet. The client confirms block chain inclusion of Ombuds bulletins.

Mobile

Model: trust your Bitcoin peers.

The mobile client includes an SPV client with bloom filters, thus receives only relevant Bitcoin transactions and every Bitcoin block header. Trusting your Bitcoin peers to honor bloom filters reduces total bandwidth usage and recording only block headers reduces total disk space, yet introduces doubt in complete message transparency. Unlike the desktop client, the mobile client does not have an index of all Ombuds statements. These compromises are necessary for the convenience of mobile devices.

The mobile client has a full featured Bitcoin wallet and an Ombuds message formatter, also presented to the user in a familiar interface. The mobile client confirms block chain inclusion of Ombuds bulletins, but again reliant on peers honoring bloom filters. Peers, in practice, do honor bloom filters.

Web Relay

Model: trust someone else.

Someone running an Ombuds Desktop Client possesses an index of all Ombuds statements. This index can be formatted and delivered as a web page, viewable to anyone with a browser and an [uncensored] Internet connection. However, the person viewing Ombuds statements must completely trust the web relay to not censor content. This option provides ultimate convenience: zero disk space, near zero processing power, and near zero bandwidth to see Ombuds bulletins, but lost is provable transparency of bulletins.
